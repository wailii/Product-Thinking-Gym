# Session 记录：助手发布与共享机制优化

## 📅 会话概览 (Session Overview)
- **日期**: 2025-12-24
- **主题**: 解决自建助手无法在公共区展示和共享的问题
- **对抗轮数**: 3（CTO + VC + User）
- **最终信心指数**: [待定]/10

---

## 1. 提案快照 (The Pitch)

> 现状：客户用 RAG 创建了助手，但只有创建者账号在深层菜单能看到。其他员工用不了，部署后交付体验极差。

| 维度 | 内容 |
|---|---|
| **一句话** | **增加“发布/上架”功能，允许用户将自建助手从“个人空间”推送到“公共市场/工作台”，供全员使用。** |
| **目标用户** | **企业交付客户**（特别是需要快速配置业务助手的管理员和业务专家）。 |
| **核心价值** | **把“单机版”变成“联机版”**。让配置好的 RAG 助手能立即被业务人员使用，无需开发介入。 |
| **商业模式** | **提升交付满意度（减少退款/投诉）**，长期可作为“企业版”的高级权限功能（按共享范围收费）。 |

## 2. 对抗实录 (The Battle Log)

### Round 1: 技术可行性 (CTO)
> "技术上能做，但值得做吗？"

- **CTO 质疑**: 
  1. **权限隔离灾难**：现在是个人私有，改成公开，知识库的权限怎么算？如果我用财务部的文档建了个助手，发布出去，是不是全公司都能看财报了？
  2. **垃圾场效应**：没有审核机制吗？如果 100 个员工每人建 10 个测试助手全点发布，公共区直接废掉，怎么检索？怎么排序？
  3. **版本地狱**：创建者修改了提示词或知识库，正在用的其他人是实时生效还是需要重新发布？如果改崩了怎么回滚？
  4. **资源爆炸**：以前私有没几个人用，现在全员用，Token 消耗和向量检索压力会不会指数级上升？

- **我的防守**: 
  1. **权限控制**：不是全员发布。**仅允许“超管”角色创建并发布面向全员的助手**。普通员工创建的依然是私有的。这样直接规避了敏感数据泄露风险，因为超管有责任确保知识库安全。
  2. **垃圾场治理**：因为只有超管能发布，所以公共区不会有垃圾助手。普通员工的测试助手都在自己的私有空间，不干扰公共区。
  3. **版本更新**：修改后，**新开启的会话生效**，旧会话保持原样。不搞复杂的版本回滚，依靠超管的人工管理（改崩了就改回来）。
  4. **资源策略**：目前用户活跃度低，**先不加限流，先把量跑起来**。等并发真上去了再考虑技术优化。

- **结论**: ✅ 通过 (方案简化：通过收敛“发布权限”到超管，解决了大部分技术风险)

### Round 2: 商业逻辑 (VC)
> "Why Now? Why Us? How Much?"

- **VC 质疑**: 
  1. **Why Now (是伪需求吗？)**: 活跃度低是因为找不到助手，还是因为助手本身烂？
  2. **Why Us (竞争力)**: 竞品都有发布到钉钉/飞书，你只做个发布到广场，够打吗？
  3. **How Much (收益)**: 怎么衡量成功？是用户活跃度还是什么？

- **我的防守**: 
  1. **场景特殊性**：客户是**电厂**，内网隔离环境，**数据安全是第一刚需**。他们用不了公网竞品，只能用我们。所以“活跃度低”不是因为产品烂，而是因为新部署且还没开始推。
  2. **无需过度集成**：电厂数字化程度有限，暂时不需要集成到 OA/飞书。现阶段只要能在我们的平台上方便地找到助手即可。
  3. **衡量指标**：**交付效率提升**。成功的标准是：后续再有同类“能不能把这个助手公开”的需求，**无需前后端开发介入改数据库或代码，交付/运维人员在后台点一下就能完成**。这就是巨大的成本节约。

- **结论**: ✅ 通过 (明确了“交付提效”和“内网独占”的核心逻辑)

### Round 3: 用户价值 (User)
> "我为什么要换？"

- **User 质疑**: 
  1. **识别官方**: 我怎么知道哪个是正规的？
  2. **便捷访问**: 能不能收藏？别让我每次都去大海捞针。
  3. **下架感知**: 突然没了怎么办？直接 404 很吓人。

- **我的防守**: 
  1. **展示逻辑**: 超管发布的助手，直接**并入“通用助手”列表**，享受同等待遇。对于用户来说，不需要区分“这是官方预置的”还是“这是公司后加的”，只要好用就行。
  2. **收藏功能**: **接受建议，加入收藏功能**。允许用户把高频使用的助手固定在首页或侧边栏。
  3. **下架处理**: 不会直接消失。如果是维护或下架，**点击时会弹出友好提示**（如“该助手正在维护中”），避免用户产生恐慌或以为系统坏了。

- **结论**: ✅ 通过 (补充了收藏和友好提示的交互细节)

## 4. 训练总结 (Post-Mortem)

### 💡 核心洞察 (Key Insights)
- [x] **“超管发布”是 ToB 复杂权限的平替**：在内网隔离、低并发场景下，不要搞复杂的 RBAC 或审核流，直接用“超管独裁”最简单有效。
- [x] **交付效率也是 ROI**：商业价值不一定是赚多少钱，省下反复修改数据库的运维成本，就是最直接的利润。
- [x] **体验闭环**：不仅要“能用”，还要“能找”（收藏）和“能知”（维护状态），这是避免一线员工恐慌的关键。

### 🔴 暴露的盲区 (Blind Spots)
- [ ] **用户感知**：最初忽略了用户如何区分官方/非官方助手，以及下架时的恐慌心理。
- [ ] **版本管理**：最初未考虑版本更新时的用户体验，后确定为“新会话生效”。

### 🟢 掌握的技能 (Mastered Skills)
- [x] **Red Teaming 防守**：成功抵御了技术、商业、用户三方的核心质疑。
- [x] **场景化剪裁**：针对“电厂内网”场景，果断砍掉了“多级审核”、“集成 OA”等冗余需求。
